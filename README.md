# Frontend Engineer Challenge

Frontend проект auth-флоу Orbitto на основе
[дизайна](https://www.figma.com/design/31KetUbya482vMSGgyiNIf/Orbitto-%7C-Service--Copy-?node-id=102-12806&t=TMlkJ3c3j3vJF5fb-4) под реальный [backend fork](https://github.com/aiezq/engineer-challenge) для 3 сценариев:

- Регистрация
- Авторизация
- Восстановление пароля

## Что реализовано

- сценарии регистрации / авторизациии / djcстановления доступа
- route guards: если уже вошли, то не можем перейти на авторизацию, если не вошли, не можем войти на приватные страницы
- интеграционные тесты
- генерация api контрактов

## Запуск

Запускаем скрипт init.sh, который соберет образы и запустит их, ресурсы доступны по ссылкам

```bash
  sh ./config/scripts/init.sh
```

### Доступные endpoints:

- **Backend**: `http://localhost:4443/graphql`
- **Frontend**: `http://localhost:4443`
- **Database**: `localhost:5432` (PostgreSQL)
- **Redis**: `localhost:6379`

## Trade-offs: Архитектурные решения проекта

### SPA or Framework with SSR

Выбор сделан в пользу SPA, так как подразумевается сложный инструмент с большим количеством интерактивных элементов.

- **Жертвуем:** SEO _(контент не индексируется поисковиками)_, временем первой загрузки _(initial bundle + data fetching)_
- **Получаем:** Более отзывчивый интерфейс после загрузки, отсутствие задержек на серверных перерендерах, более простой деплой _(только статика)_

### Default or FSD or ED

Выбран **ED (Evocomm Design)** — архитектура, основанная на модулях и гибких слоях, без избыточной жёсткости FSD.

Жертвуем: Огромным сообществом и готовыми примерами FSD, привычной структурой со слоями pages/widgets/entities, строгими ограничениями, которые дисциплинируют команду
Получаем: Гибкость _(можно начинать с одного файла и усложнять модуль по мере роста)_, отсутствие оверхед-абстракций на старте, явное разделение на features _(основная логика)_ и services _(переиспользуемые бизнес-модули)_, возможность хранить бизнес-логику в shared _(в отличие от FSD)_, поддержку подмодулей и групп, 4 готовые модификации _(от small до monorepo)_

### React or Vue

React даёт архитектурную свободу, идеальный TypeScript через JSX, огромную экосистему и путь к мобильной разработке через React Native.

- **Жертвуем:** Чуть большим бандлом _(\~40kb против \~20kb)_, отсутствием нативной анимации переходов (нужны сторонние библиотеки), менее стабильными DevTools (иногда врут), более крутой кривой обучения для новичков
- **Получаем:** Многообразие инструментов под любую задачу, строгую типизацию без магии, возможность переиспользовать логику на React Native, полный контроль над архитектурой

### React Router or TanStack Router

Выбран **React Router** из-за большей зрелости и лучшей совместимости с React Compiler _(при билде компоненты разбиваются на мелкие JS-файлы — с TanStack Router такой оптимизации не сработало)_.

- **Жертвуем:** Встроенной типизацией поисковых параметров _(search params)_ на уровне TypeScript, более строгой типобезопасностью маршрутов, параллельными загрузчиками данных
- **Получаем:** Проверенный годами роутер с огромным сообществом, стабильную работу с React Compiler _(автоматическое code-splitting)_, отличную документацию, простую миграцию с легаси-кода

### Fetch or Apollo Client or TanStack Query or Redux Toolkit (Queries)

Выбран **TanStack Query** для работы с серверным состоянием. GraphQL-запросы генерируются через `@graphql-codegen`, создавая типизированные хуки.

- **Жертвуем:** Встроенным кэшем Apollo Client _(но TanStack Query с codegen даёт достаточно)_, автоматической нормализацией GraphQL-ответов, встроенным UI для отладки запросов _(Apollo Studio)_
- **Получаем:** Агностицизм к транспорту _(легко переключиться с GraphQL на REST без переписывания логики кэширования)_, простую инвалидацию и повторные запросы, дедупликацию запросов, удобную работу с пагинацией и бесконечной прокруткой, меньший размер бандла, полный контроль над кэшем

### React Hook Form or TanStack Form

Выбран **TanStack Form** как более современное и типобезопасное решение, а также для единообразия с экосистемой TanStack _(Query уже используется)_.

Жертвуем: Огромной экосистемой и проверенной годами стабильностью **React Hook Form**, бóльшим количеством готовых примеров и интеграций
Получаем: Первоклассную TypeScript-поддержку _(вывод типов без бойлерплейта)_, декларативный API, отличную работу с вложенными и динамическими полями, единую ментальную модель с **TanStack Query**

### Zod or Yup

Выбран **Zod** для валидации схем на клиенте.

- **Жертвуем:** Меньшим количеством готовых примеров в легаси-коде (Yup старше), отсутствием встроенной локализации ошибок _(нужны обёртки)_
- **Получаем:** Первоклассную поддержку TypeScript _(вывод типов из схемы без дублирования)_, более производительный парсинг, отличную композицию схем (`.merge()`, `.pick()`, `.extend()`), детализированные ошибки с путями к полю

### Tailwind or CSS Modules or Styled Components

Выбран **Tailwind CSS** как утилитарный CSS-фреймворк.

- **Жертвуем:** Чистотой HTML _(много классов в разметке)_, разделением стилей и логики _(стили прямо в компонентах)_
- **Получаем:** Беспрецедентную скорость разработки _(не нужно придумывать имена классов)_, отсутствие переключения контекста _(не надо открывать отдельный CSS-файл)_, автоматическую очистку неиспользуемых стилей _(PurgeCSS)_, маленький итоговый CSS-бандл

### Vite or Webpack

Выбран **Vite** как сборщик для разработки и продакшен-билда.

- **Жертвуем:** Огромной экосистемой плагинов Webpack _(но Vite уже догнал по основным сценариям)_, максимальной кастомизируемостью на грани извращений
- **Получаем:** Мгновенный запуск dev-сервера _(даже на больших проектах)_, мгновенный HMR _(горячая замена модулей)_, прозрачную конфигурацию _(конфиг не спрятан в недрах — ты его видишь и контролируешь)_, отличную скорость сборки на продакшене, меньше боли с настройкой

### ESLint + Prettier or Biome

Выбрана связка **ESLint + Prettier** из-за зрелости.

- **Жертвуем:** Скоростью работы _(ESLint + Prettier медленнее Biome)_, одной зависимостью вместо двух, простотой конфигурации _(у Biome всё в одном файле)_
- **Получаем:** Огромную экосистему правил _(eslint-plugin-import, eslint-plugin-unicorn, eslint-plugin-jest-dom и сотни других)_, проверенную годами стабильность, поддержку любых экспериментальных фич JS/TS _(ESLint обновляется быстрее)_, максимальную гибкость кастомизации

### Zustand or Redux or Jotai

**Выбор ещё не окончательный.** На данный момент есть предварительная склонность к **Jotai** из-за минималистичности и соответствия всем текущим требованиям.

- **Жертвуем:** Cтрогим паттерном "один источник правды" на всё приложение _(атомы могут быть распределёнными)_, огромным сообществом Redux, готовыми архитектурными паттернами (Ducks, slices)
- **Получаем:** Минималистичный API (учится за 10 минут), атомарную модель _(перерендер только тех компонентов, которые подписаны на конкретный атом, без лишних обновлений)_, отличную TypeScript-поддержку _(вывод типов из коробки)_, лёгкую композицию (атомы можно строить из других атомов), нулевой бойлерплейт, встроенную поддержку асинхронных и зависимых состояний (`atom(async read => ...)`)

Выбираем **Jotai**, так как:

1. Проект не требует глобальной строгой централизации (Redux — overkill)
2. Атомарная модель идеально ложится на композицию UI-состояний
3. Jotai легко интегрируется с TanStack Query _(можно синхронизировать атомы с данными из кэша)_
4. Размер бандла критичен для SPA с тяжёлой интерактивностью

## Cледующиe шаги в production-версии

### frontend

- стандартизация ошибок от сервера
  - если сценарий: бек всегда отдает ошибки на одном английском - отдельный список ошибок и обработчик
  - если сценарий: бек сам локализует ошибки - в запрос добавить параметр языка и обработчик
- настроить husky, дабы избезать ситуаций, когда заливается нерабочий код
- вынести тексты в отдельный файл
  - если планируется локализация - i18n
- e2e тесты
- проверка и отображане вида "есть ли интернет"
- документы для политики приватонсти и оферты
- документация вида "мы используем cookie"
- выбрать решение для анимаций переходов состояний компонентов
- страница профиля аккаунта
- страница дашборда

### backend

- доработать аутентификацию
  - роут для обновления токена доступа
  - success должен стать secure, same-site, http-only
  - роут для выхода
  - роут регистрации должен возвращать токен доступа для возможности сразу войти в систему
  - роут логина должен возвращать только токен
- роли для пользователя, добавить роль "неактивный пользователь" для блокировки запросов, пока пользователь не подтвердил почту
- договорится о формате ошибок
  - будет ли ответ от сервера всегда 200 или использовать стандартные коды _(400, 401,403...)_
  - разработать стандартизированный тип ошибок для обычных случаев и случаев ввода данных
